## Source Data

The private dataset build pipeline derives the public workshop dataset from the GEFCom2014 load forecasting data.
This source data is not committed to the repo and must be downloaded manually before running the build pipeline.

### Downloading GEFCom2014 from Kaggle

1. Go to [https://www.kaggle.com/datasets/cthngon/gefcom2014-dataset](https://www.kaggle.com/datasets/cthngon/gefcom2014-dataset)
2. Sign in to your Kaggle account (required to download)
3. Click **Download** to get the zip archive
4. Place the downloaded zip at:
   ```
   private/data/gefcom2014_raw.zip
   ```
5. Unzip it. The load track data lives under `load/` inside the archive, with task subfolders (`Task 1/`, `Task 2/`, ...) each containing `L{n}-train.csv` and `L{n}-benchmark.csv` files.

### Using the source data

Pass the path to the prepared source CSV or parquet to `derive_public_dataset_from_source_path()` in `private/derive_dataset.py`.
The pipeline expects columns: `timestamp`, `zone`, `demand`, `temp`.
You will need to wrangle the raw GEFCom2014 CSVs into that shape before passing them to the build function.

The `private/data/` directory is not tracked by git and should not be committed.

## Packaging Workflow

The private repo assembles a ready-to-publish snapshot into a separate local checkout of either the staging repo or the public repo.

The packaging script must:
1. Validate the target workshop repo checkout
2. Confirm the destination is not this dev repo and that its name or `origin` remote matches `RichardPovinelli/SGA_workshop_test` or `RichardPovinelli/SGA_workshop`
3. Delete everything except `.git`
4. Copy only approved public files
5. Stop before any git push, tag, or release step

## Release Rehearsal

After staging release-ready external assets, run the local rehearsal before any manual publication step:

```bash
python tools/rehearse_public_release.py
```

The rehearsal rebuilds the configured local checkout with the packaging script, validates the packaged repo shape, runs the public checkout tests, checks notebook cleanliness, validates staged dataset and TFT assets from `release_assets_manifest.json`, and exercises the runtime install helpers against a local-safe state root.

The default report is written to `reports/public_release_rehearsal.md`.
It ends with either `READY FOR MANUAL PUBLICATION` or `NOT READY FOR MANUAL PUBLICATION`.
Publication remains manual; the rehearsal does not commit, push, tag, or create GitHub releases.

Use `SGA_workshop_test` as the first full staging target. Set `SGA_WORKSHOP_PUBLIC_REPO_PATH` to the local staging checkout, rehearse, publish test releases there, validate bootstrap from that repo, and only then repeat the same process against `SGA_workshop`.

Before running the rehearsal for final review:
- set `SGA_WORKSHOP_PUBLIC_REPO_PATH` or `config/paths.py` to the separate local `SGA_workshop_test` or `SGA_workshop` checkout
- stage both the dataset `.parquet.gz` asset and TFT `.tgz` bundle under the configured release staging root
- ensure `release_assets_manifest.json` includes matching filenames, release tags, compatibility versions, expected install paths, and `sha256` checksums

## Data Format Contract

The public runtime is expected to read parquet datasets using the `pyarrow` backend through pandas.
That backend choice is intentional so fresh environments do not silently miss parquet support.
Tests in this repo are expected to exercise a real parquet round trip, including the `.parquet` and `.parquet.gz` paths planned for the public loader.

## Dataset Build Acceptance

The released workshop dataset is expected to be generated by the private build pipeline, not maintained as a hand-edited artifact.
It is expected to be distributed as a GitHub release asset rather than committed to the repo as a version-controlled dataset file.
Task completion for the dataset pipeline requires fixed-seed reproducibility checks and a machine-readable build manifest for the emitted public dataset.
That manifest is expected to record the artifact name, row count, column list, zone list, date range, split counts, build seed, and build timestamp.
The transform policy must also be documented clearly enough that maintainers can verify why the released data is reproducible while still not being a trivial one-to-one recovery of the private source series.

## Bootstrap Artifact Contract

The bootstrap and release-rehearsal steps are expected to share one concrete dataset artifact contract.
That includes a stable dataset filename convention, one shared tag-based GitHub asset-resolution contract, and one predictable local install path outside the public repo root.
The intended config shape is based on repository identity plus tag plus filename, for example `GITHUB_REPO_OWNER`, `GITHUB_REPO_NAME`, `DATASET_RELEASE_TAG`, and `DATASET_ASSET_FILENAME`.
Release rehearsal should exercise that same bootstrap contract with a local-safe artifact source when the final published asset is not yet live, rather than inventing a separate rehearsal-only path.

## TFT Artifact Contract

TFT inference is part of the standard public workshop environment by default.
The TFT checkpoints are expected to come from a separate GitHub release stream from the dataset, using one bundled `.tgz` artifact that covers horizons `1..7`.
Bootstrap and release rehearsal should share one explicit TFT tag-based bundle contract, install the extracted checkpoints under a predictable local path outside the public repo root, and keep checkpoint download or extraction logic out of notebook cells.
The intended config shape is based on repository identity plus tag plus filename, for example `TFT_RELEASE_TAG` and `TFT_BUNDLE_FILENAME`.
The TFT release tag may differ from the dataset release tag and should be resolved independently.
The extracted bundle is expected to follow one canonical installed layout, including a stable manifest or checkpoint-map filename and relative checkpoint paths resolved from the extracted TFT root.

## Release-Ready Asset Staging

Before any later manual publication step, the private repo workflow is expected to stage the exact external assets that will be uploaded to GitHub releases.
That staged set should include the final dataset artifact, the final TFT checkpoint `.tgz` bundle, and machine-readable metadata describing filenames, versions, compatibility metadata, and expected local install paths.
Release rehearsal should validate that same staged asset set rather than a separate rehearsal-only shape.

## Supported Environments

The standard public workshop path is designed for Google Colab and POSIX-style shell environments.
WSL is the expected local verification environment on Windows machines and should be treated as the local POSIX-compatible test path.
Native Windows shell execution is not part of the guaranteed public workshop contract unless explicitly added later.

## Notebook Cleanliness

Notebook files stored in the repo are expected to be clean and output-free.
Execution outputs should be stripped before committing so the repo contains source notebooks rather than cached execution state.
Reusable plotting, formatting, and reporting helpers should live in `src/`, while notebook-local one-off display code should remain lightweight and non-reusable.
Transient notebook metadata should be kept minimal where practical so notebook diffs stay readable.

## Canonical Public Config

Public runtime code, bootstrap setup, notebook orchestration, and release rehearsal are expected to share one canonical config surface under `src/`.
That config should be the source of truth for dataset paths, TFT artifact paths, release tags, asset filenames, and the installed TFT manifest/checkpoint-map filename.
Environment variables may be used for controlled overrides during testing or rehearsal, but they should override that canonical config rather than replace it.
The canonical config surface may begin in `src/config.py` and later evolve into a small config package if that improves clarity, while still remaining the single source of truth.

## Results Schema

Model evaluation outputs are expected to share one common base results-table contract across baselines, XGBoost, and TFT comparisons.
Later model-specific outputs may extend that schema, but should not redefine the common fields such as zone, horizon, model, and core metrics.

## Zone Contract

Zone identifiers are expected to be stable strings with deterministic ordering across manifests, model outputs, and notebook presentation.
That keeps per-zone comparisons and workshop demos consistent across the workflow.

## Asset Compatibility

Because the dataset asset and TFT checkpoint bundle can use independent release tags, the public runtime is expected to validate compatibility metadata before treating installed assets as usable.
The canonical public config should declare the expected dataset compatibility/schema version and TFT checkpoint-format or runtime-compatibility version, and bootstrap plus rehearsal should validate those expectations explicitly.
The plan now assumes exact field-level compatibility manifests rather than a loose “compatible enough” check.

## Bootstrap Rerun Safety

Bootstrap is expected to be safe to rerun.
It should detect what is already installed, avoid mixing old and new TFT bundle contents, handle stale or partially installed assets explicitly, and record install-state metadata describing what dataset and TFT artifacts are currently installed.
That install-state is expected to live only in local state directories outside the public repo root and must not be treated as packaged public source metadata.

## Metadata Boundary

The packaged public repo should contain only the public-runtime metadata needed for bootstrap, runtime validation, and notebook orchestration.
Staged upload manifests, unpublished external assets, and private release-preparation metadata should remain outside the packaged public repo and be validated only through the external-asset staging and rehearsal flow.
The plan now distinguishes between the clean packaged repo shape before bootstrap and the expected local-state shape after bootstrap.
Persistent post-bootstrap local asset directories and install-state files are expected to live outside the public repo root so setup does not create noisy tracked changes inside the public checkout.

## Staged Release Manifest

Release-ready external assets are expected to be described by one stable staged manifest such as `release_assets_manifest.json`.
That manifest should include both dataset and TFT asset entries, with filenames, release tags, install paths, and compatibility metadata, and rehearsal should treat it as the source of truth for staged asset validation.
